Data Privacy And Permissions

This document explains how data is stored and accessed with the @ protocol. There are many things that are quite different from previous models as a result of strict adherence to data privacy principles. For clarity, it is written from the perspective of how your application works on someone’s device, who is the owner of their data, controls how their data is shared with others and how accessing data that others have shared with them honors privacy controls for that data. Control of shared data always accrues to the benefit of the entity that owns it. This applies to data that is shared by the owner with other people as well as all data that others have shared in return.

WARNING: Data access is very different with the @ protocol than what most developers are used to.

Application Access to Data

There are four basic types of application data access and you need to become familiar with how privacy is ensured for each:

  1. Storing and retrieving data for my application only
  2. Sharing data between entities
  3. Peer-to-Peer applications data sharing
  4. Access to data stored by other applications

Storing and retrieving data for my application only

If your application just needs to store data and have it available across all the devices that a person owns, you can manage data securely and privately using the @persistence keystore. This is easy to do using the @Client SDK and flutter widgets library with the following features:

  • All data for your application is encrypted
  • In-memory access is super fast
  • Data is synchronized across all the person’s devices automatically
  • Privacy is ensured as only the owner of the data can interact with it

For more information on how to use this, see the @Persistence keystore guide and the @Client Cookbook

Sharing data between entities

Peer-to-Peer applications data sharing

Applications typically are a combination of my own data as well as data that is shared with me by others.  For example, a chat application combines messages that I share with one or more people as well as messages that they share with me. The chat application is then responsible for interleaving these messages while simultaneously maintaining privacy controls as per the owner of the data (ii.e. I own my messages and you own yours).

While this may seem confusing and difficult at first, the @Client SDK and the Flutter UI Component libraries make it very simple.

Interacting with data shared with me by others

Applications can reason over all data in my @persistence keystore (@certification required)

One super interesting side effect of giving people control of their data and storing it all in one place is that any application that they choose can reason over any data that they are allowed to access in order to create altogether new user experiences. Applications that are certified as @ Protocol compliant can create amazing new user experiences with the ability to access and reason over all data stored in an @keystore.

For example, their @ compliant messaging application may contain a thread where a group of people are discussing which movie to go see on Wednesday night. If permitted, this information can also be presented as an event in their @ compliant calendar application as well as presented as a group in their @ compliant contacts application.

Kinds of access for data in the @persistence keystore

Publicly shared data: is, you guessed it, publicly accessible and can be read by anyone with a simple @protocol lookup for the given @sign.

To help to make this clear, we will use a simple example:

lookup:location@kevin => “Silicon Valley” // public value set by @kevin

Privately shared data: is only accessible by the owner and the individual designated recipient using @protocol lookup after completing the pol: handshake for the given @sign.

So, in this case, the same lookup can return a different result.

pol:@colin

lookup:location@kevin => “1900 Camden Ave” // private value for @colin set by @kevin

Note: each private setting is stored as an individual record so that access control is “hard coded” to make it really easy to ensure robust privacy controls.

Creating, updating and deleting data: can only be done by the owner of the data as controlled by access to the secret key for their data.

Note: the cloud microservice does not have access to the secret key to prevent bad actors from accessing or modifying private data not meant for them.

Access within an application namespace

Applications that only rely on data within its own namespace can also store data in the @keystore if certified as @ Protocol compliant. If the data being stored rightfully belongs to the person creating it (which is the case most of the time), then they will have control of how it is used and shared with other entities and applications.

Sharing data across applications

One of the primary advantages of the @protocol is that your application may have (with permission, of course) the ability to “reason” over all of the data stored across all apps. This may seem non-intuitive at first, but since the owner controls access to their data, it is up to them what data your app is allowed to access.
